# 📘 7차시 — 큐(Queue) · 데크(Deque) · 우선순위 큐(Priority Queue)

> 핵심 한 줄 요약
> 큐는 FIFO 구조로 Rear에서 삽입·Front에서 삭제. 선형 큐는 가짜 포화 문제가 있어 원형 큐로 해결. 데크는 양쪽 모두 삽입·삭제 가능(이중 연결 리스트로 구현). 우선순위 큐는 이름은 큐지만 힙(Heap, 트리 기반) 으로 구현해야 효율적. 이 차시는 기말고사 족집게 출제 파트!

## 1. 큐 추상 데이터 타입 (Queue ADT)

### 1.1 큐(Queue)의 기본 정의
- 순서 있는 리스트 (Ordered List)
- FIFO (First In First Out) 구조: 스택(LIFO)과 달리 먼저 들어간 데이터가 먼저 나옴.
    - 🗣️ 교수님 비유: 버스의 타는 곳과 내리는 곳이 따로 있어 먼저 탄 사람이 먼저 내리는 것과 같음.

### 1.2 주요 연산 위치

| 연산 | 위치 | 이름 |
|------|------|------|
| 삽입 (Enqueue) | 큐의 한쪽 끝 | rear (후단) |
| 삭제 (Dequeue) | 반대쪽 끝 | front (전단) |

---

## 2. 큐의 정적 구현: 선형 큐(Linear Queue)와 그 한계

### 2.1 선형 큐의 동작 메커니즘

- 배열 사용, 크기 고정, 연속적 메모리.
- 초기값: `front = -1`, `rear = -1`
- 삽입: `rear`를 1 증가시키고 그 위치에 데이터 넣기
- 삭제: `front`를 1 증가시키고 그 위치의 데이터 삭제(반환)

---

### 2.2 💡 선형 큐의 치명적인 문제점 (가짜 포화 상태)

#### 문제 상황
- 삽입·삭제를 반복하다 보면 `front`와 `rear`의 값이 계속 증가만 함.
- 결국 `rear`가 `MAX_QUEUE_SIZE - 1`에 도달하면, 앞부분(front 쪽)에 삭제로 인해 빈 공간이 생겼음에도 불구하고 더 이상 데이터를 삽입할 수 없는 상태 (가짜 포화 상태) 발생.

#### 시각화
```
[_, _, _, D, E, F]   ← rear가 끝에 도달, 앞이 비어있지만 삽입 불가!
 ↑           ↑
 빈칸        rear = 5 (MAX-1)
```

#### 🗣️ 교수님 질의응답
- 학생 질문: "딜리트 때 프론트 인덱스가 안 움직이는데... `-1, -1` 이렇게 가는 건 맞나요?"
- 교수님 답변:
    - 데이터를 삭제할 때마다 프론트 값을 증가시켜야 함.
    - 만약 인덱스를 움직이지 않고 빈자리를 채우려 한다면 마치 자료를 이동(Shift)한 것 같은 느낌을 줘야 함.
    - → 해결하려면 남은 데이터를 앞으로 당기는 Left Shift 연산 필요.
    - 하지만 "자료의 이동을 하고 싶지 않다(비효율적이다)" 는 것이 핵심 딜레마.

→ 해결책: 원형 큐 등장!

## 3. 원형 큐 (Circular Queue) 🌟

선형 큐의 '데이터 이동(Shift)' 문제를 해결하기 위해 고안된 개념.

### 3.1 원형 큐의 핵심 개념

- 배열을 마치 원형(Circular) 인 것처럼 다루어, 끝까지 가면 다시 처음으로 돌아가게 만듦.
- 나머지 연산자(Modulo, `%`) 사용: 인덱스를 계산할 때 `MAX_QUEUE_SIZE`로 나눈 나머지를 사용하여 배열의 처음으로 돌아감.

#### 시각화
```
       [0]
    [5]   [1]
   [4]     [2]
       [3]
        ▲
    (front, rear가 돌면서 이동)
```

---

### 3.2 💡 원형 큐의 초기 설정 및 상태 조건 (매우 중요, 시험 출제 포인트) 🌟

| 항목 | 조건 | 비고 |
|------|------|------|
| 초기값 | `front = 0`, `rear = 0` | ⚠️ 선형 큐처럼 `-1`이 아님! |
| 최대 저장 개수 | `MAX_QUEUE_SIZE - 1` 개 | 전체 크기 5라면 최대 4개까지만 저장 |
| Empty (공백) 조건 | `front == rear` | front와 rear가 같은 위치 |
| Full (포화) 조건 | `(rear + 1) % MAX_QUEUE_SIZE == front` | rear에 1을 더한 위치가 front와 같을 때 |

#### 🔑 왜 한 칸을 비워두는가?

> Empty와 Full 상태를 구분하기 위해서이다.
> 비워두지 않으면 둘 다 `front == rear`가 되어 구분이 불가능해짐.

---

### 3.3 원형 큐의 연산 과정 (교수님 예시 상세 트레이싱)

전제: 배열 크기 6 (0~5), 최대 5개 저장 가능.
초기 상태: `front = 0`, `rear = 0`

#### Enqueue (삽입) 연산
```c
rear = (rear + 1) % MAX_QUEUE_SIZE;
queue[rear] = data;
```

#### Dequeue (삭제) 연산
```c
front = (front + 1) % MAX_QUEUE_SIZE;
return queue[front];
```

#### 예시 trace

| 연산 | front | rear | 내용 |
|------|-------|------|------|
| 초기 | 0 | 0 | 비어있음 |
| `in_q(J1)` | 0 | 1 | rear 1 증가, J1을 queue[1]에 삽입 |
| `in_q(J2)` | 0 | 2 | J2를 queue[2]에 삽입 |
| `dq()` | 1 | 2 | front 1 증가 → J1 삭제 |

#### 💡 특징 (교수님 강조)
- 선형 큐에서는 항상 `rear >= front` 였지만, 원형 큐에서는 계속 회전하기 때문에 `front`가 `rear`보다 더 큰 값을 가질 수도 있음.

## 4. 큐의 동적 구현 (연결 리스트 활용)

정적 배열의 크기 제한을 극복하기 위해 연결 리스트를 사용하여 동적으로 구현.

### 4.1 동적 큐의 기본 구조

| 포인터 | 역할 |
|--------|------|
| `front` | 연결 리스트의 맨 앞 노드를 가리킴 (삭제/Dequeue와 관련) |
| `rear` | 연결 리스트의 맨 뒤 노드를 가리킴 (삽입/Enqueue와 관련) |

#### 💡 Empty 조건
- `front == NULL` (또는 `front`와 `rear`가 모두 NULL)
- 동적 구현에서는 크기 제한이 없으므로 Full 상태를 체크할 필요가 없음.

### 4.2 삽입 (Enqueue) 연산 메커니즘

> 리스트의 `append` 개념과 동일. 맨 마지막에 노드 추가.

```
① 새 노드(temp) 생성 및 데이터 설정
② temp->link = NULL  (마지막이 될 것이므로)
③ rear->link = temp  (기존 rear의 링크를 새 노드로)
④ rear = temp        (rear 포인터를 새 노드로 업데이트)
```

### 4.3 삭제 (Dequeue) 연산 메커니즘

> 삭제할 노드(맨 앞 노드)를 먼저 설정.

```
① 공백 검사: front == NULL이면 삭제 불가
② temp = front       (임시 포인터에 front 저장)
③ front = front->link  (front를 다음 노드로 이동)
④ 데이터 반환 및 free(temp)
```

## 5. 데크 (Deque, Double-Ended Queue)

### 5.1 데크의 개념
- 큐의 전단(front)과 후단(rear) 양쪽 모두에서 삽입과 삭제가 가능한 큐.
- 스택과 큐의 장점을 합친 형태.

> 🗣️ 발음 주의 (교수님 언급): "Neck"이 아니라 "데크(Deque)" 라고 발음!

---

### 5.2 💡 데크의 동적 구현 (이중 연결 리스트)

#### 왜 이중 연결 리스트(Doubly Linked List)를 사용하는가?
- 양방향에서 삽입/삭제를 하려면 `front`와 `rear` 각각에 대해 삽입용/삭제용 인덱스를 관리해야 하므로 정적 배열로는 매우 불편.
- 따라서 `LLink`(왼쪽 링크)와 `RLink`(오른쪽 링크) 를 가지는 이중 연결 리스트를 사용하는 것이 일반적.

#### 포인터 구성
- `head` : front 역할
- `tail` : rear 역할

---

### 5.3 💡 [Q&A 하이라이트] 데크의 삭제 연산과 메모리 해제(Free) 순서 논쟁

> 교수님과 최경찬 학생 간의 아주 구체적인 질의응답. 시험에 로직 순서 문제로 출제될 수 있음.

- 상황: `deleteRear` (맨 뒤 노드 삭제) 연산 중.
- 학생의 질문: "지울 노드를 굳이 `removed`라는 임시 포인터에 담아두지 않고, 그냥 `free(tail)` 해버리면 안 되나요?"
- 교수님의 답변 및 로직 정리:

| 단계 | 동작 |
|------|------|
| ① 임시 포인터의 필요성 | 삭제할 노드를 누군가(임시 포인터 `removed` 또는 `temp`)는 가리키고 있어야 함 |
| ② 연결 끊기 | `tail = tail->LLink` (tail 포인터를 앞 노드로 이동) |
| ③ 새로운 마지막 노드 설정 | 이동한 tail 노드의 `RLink`를 NULL로 만들어 줌 |
| ④ 메모리 반환 | 아까 `removed`에 담아둔 노드를 `free` 시킴 |

#### 🔑 핵심 결론
> 기존 정보를 지우기 전에 연결을 먼저 안전하게 옮겨놓아야 한다.
> 다만, 임시 포인터에 담아두었다면 메모리 반환(`free`) 자체는 연결 리스트 구조를 망치지 않는 선에서 순서가 유연해질 수 있음.

## 6. 우선순위 큐 (Priority Queue)

선형 자료구조의 마지막이자, 비선형 자료구조(트리)로 넘어가는 다리 역할을 하는 매우 중요한 개념.

### 6.1 우선순위 큐의 핵심 개념

- 정의: 일반적인 큐의 FIFO 순서가 아니라, 우선순위가 높은 데이터가 먼저 나가는(Dequeue) 큐.
- 기준: 구현에 따라 가장 우선순위가 높은 것(Highest) 이 먼저 나갈 수도 있고, 반대로 가장 낮은 것(Lowest) 이 먼저 나갈 수도 있음.

> 🗣️ 교수님 비유 (1차시): 문서를 잔뜩 프린트 걸어놨는데 사장님이 오셔서 "내 문서 먼저 뽑아!" 하시면, 사장님 문서의 우선순위를 높여서 새치기(먼저 출력) 하게 만드는 시스템.

---

### 6.2 우선순위 큐의 응용 분야

| 분야 | 용도 |
|------|------|
| 운영체제(OS) | 작업 스케줄링(Job Scheduling), 인터럽트 핸들링 (가장 긴급하고 Critical한 프로세스를 먼저 처리) |
| 시뮬레이션 시스템 | 대개 '사건의 시각' 이 우선순위가 됨 |
| 네트워크 | 트래픽 제어 |

---

### 6.3 💡 [시험 출제 포인트] 우선순위 큐의 구현 방법

#### 선형 자료구조의 한계
> 🗣️ 교수님: "너무 기쁘게도 선형 자료로는 얘는 구현을 할 수가 없어요"

- 배열이나 연결 리스트로 구현하면 삽입/삭제 시 효율성이 매우 떨어짐 (매번 우선순위 비교해야 하므로).

#### 실제 구현: 트리(Tree)와 힙(Heap) ⭐
- 우선순위 큐는 비선형 자료구조인 '트리(Tree)', 그중에서도 '힙(Heap)' 이라는 구조를 사용하여 구현하는 것이 일반적.

| 종류 | 특징 |
|------|------|
| Max Heap (최대 힙) | 가장 우선순위가 높은(큰) 아이가 루트 노드에 올라가는 구조 |
| Min Heap (최소 힙) | 가장 작거나 우선순위가 낮은 아이가 루트 노드에 가는 구조 |

#### 💡 자료구조의 함정 (1차시 재확인)
- 이름은 '큐(Queue)'지만, 비선형 자료구조(트리/힙)로 구현해야 효율적.
- 힙 정렬(Heap Sort) 과도 밀접하게 연결됨.

## 7. 스택(Stack)과 큐(Queue) 총정리 및 응용

### 7.1 스택 vs 큐 비교표

| 구분 | 스택 (Stack) | 큐 (Queue) |
|------|-----------------|---------------|
| 구조 | LIFO (Last In First Out) | FIFO (First In First Out) |
| 삽입/삭제 위치 | 한쪽 끝 (top) 에서 모두 발생 | 삽입은 rear, 삭제는 front (반대쪽) |
| 주요 응용 | 함수 호출, 수식 변환/계산, 백트래킹 (실행취소, 뒤로가기) | 버퍼링, 프린터 큐, 메시지 큐, 데이터 스트리밍 |

---

### 7.2 💡 [매우 중요] 스택을 활용한 수식 계산 (교수님 별표 강조)

> 수식을 처리할 때는 '변환 단계'와 '계산 단계' 로 나뉘며, 스택에 들어가는 내용이 완전히 다름!

| 단계 | 스택에 들어가는 것 | 핵심 규칙 |
|------|-------------------|-----------|
| 수식 변환 (Infix → Postfix) | 연산자(Operator) | 새로 들어오는 연산자와 스택 안 연산자의 우선순위 비교 → 스택에 있는 연산자가 더 작아질 때까지 계속 POP |
| 수식 계산 (Postfix Evaluation) | 피연산자(Operand, 숫자) | 연산자를 만나면 스택에서 피연산자 2개 POP 하여 계산 → 결과를 다시 PUSH → EOS 시 최종값 POP |

## 8. 💡 교수님이 직접 짚어주신 기말고사 족집게 출제 포인트 & 에피소드

> 강의 마지막 Q&A 시간에 교수님께서 기말고사에 대해 아주 구체적인 팁을 주심. (김아은 씨, 소담기 씨 등 학생들의 출석을 언급하며 분위기를 풀어주심.)

### 🎯 [교수님 피셜] 무조건 나오는 기출 3가지 (별표 쫙!) 🌟

#### ① 스택의 수식 변환/계산 트레이싱 문제
> "수식을 변환할 때, 계산할 때 스택의 변화 과정들을 지난 연습 문제로 해봤었잖아요. 그런 게 똑같이 나옵니다."

#### ② 원형 큐(Circular Queue)의 상태 조건
> "도대체 Full인 거는 어떤 조건이고, Empty는 어떤 조건인지"
> → 반드시 암기:
> - Empty: `front == rear`
> - Full: `(rear + 1) % MAX_SIZE == front`

#### ③ 큐의 연산 결과 예측 문제 ("점수 드릴게요" 문제)
> "INQ, INQ, INQ를 하고 DQ, INQ, DQ를 하게 되면 최종 결과(큐의 상태나 front/rear의 위치)는 어떻게 되는지 묻는 문제. 너무 쉽죠? 점수 드릴게요."

#### 실전 연습: `INQ, INQ, INQ, DQ, INQ, DQ` 추적 예시

전제: 원형 큐, 크기 6, 초기 `front = 0`, `rear = 0`

| 순서 | 연산 | front | rear | 큐 내부 | 비고 |
|------|------|-------|------|---------|------|
| 초기 | — | 0 | 0 | `[_, _, _, _, _, _]` | Empty |
| 1 | INQ(A) | 0 | 1 | `[_, A, _, _, _, _]` | A 삽입 |
| 2 | INQ(B) | 0 | 2 | `[_, A, B, _, _, _]` | B 삽입 |
| 3 | INQ(C) | 0 | 3 | `[_, A, B, C, _, _]` | C 삽입 |
| 4 | DQ | 1 | 3 | `[_, _, B, C, _, _]` | A 반환 |
| 5 | INQ(D) | 1 | 4 | `[_, _, B, C, D, _]` | D 삽입 |
| 6 | DQ | 2 | 4 | `[_, _, _, C, D, _]` | B 반환 |

→ 최종: front = 2, rear = 4, 큐에 `[C, D]` 2개 남음.

## 🎯 7차시 최종 암기 체크리스트

### 큐 기본
- [ ] FIFO (First In First Out)
- [ ] Enqueue → rear, Dequeue → front
- [ ] 비유: 버스 탑승/하차

### 선형 큐 한계
- [ ] 가짜 포화 상태: rear는 끝, front 쪽은 비었지만 삽입 불가
- [ ] 해결: Left Shift (but 비효율) → 원형 큐

### 원형 큐 🌟 (족집게 ①)
- [ ] 초기값: `front = 0`, `rear = 0` (선형과 다름!)
- [ ] 최대 저장: `MAX_SIZE - 1`개 (한 칸 비움)
- [ ] Empty: `front == rear`
- [ ] Full: `(rear + 1) % MAX_SIZE == front`
- [ ] 한 칸 비우는 이유: Empty ↔ Full 구분
- [ ] Enqueue: `rear = (rear + 1) % MAX_SIZE; queue[rear] = data;`
- [ ] Dequeue: `front = (front + 1) % MAX_SIZE; return queue[front];`
- [ ] 원형 큐에서는 front가 rear보다 클 수도 있음!

### 동적 큐 (연결 리스트)
- [ ] Enqueue 4단계: 생성 → link=NULL → `rear->link=temp` → `rear=temp`
- [ ] Dequeue 4단계: 공백검사 → `temp=front` → `front=front->link` → free
- [ ] Full 검사 불필요 (크기 무제한)
- [ ] Empty: `front == NULL`

### 데크 (Deque)
- [ ] 양쪽(front, rear)에서 삽입/삭제 모두 가능
- [ ] 발음: "데크" (Neck 아님)
- [ ] 이중 연결 리스트 (LLink + RLink)로 구현
- [ ] `head`(front), `tail`(rear) 포인터
- [ ] Q&A: 삭제 시 임시 포인터 사용 → 연결 먼저 안전하게 → 그 후 free

### 우선순위 큐
- [ ] 정의: FIFO 무시, 우선순위 높은 것이 먼저 나감
- [ ] 선형으로 구현 불가 (비효율)
- [ ] 트리(Tree) / 힙(Heap) 으로 구현!
- [ ] Max Heap: 큰 값이 루트 / Min Heap: 작은 값이 루트
- [ ] 응용: OS 스케줄링, 시뮬레이션, 네트워크 트래픽
- [ ] 힙 정렬과 연관

### 교수님 족집게 기출 3가지 🎯🌟
- [ ] ① 스택 수식 변환/계산 트레이싱 (6차시 내용)
- [ ] ② 원형 큐의 Full/Empty 조건
- [ ] ③ 큐 연산 결과 예측 (`INQ INQ INQ DQ INQ DQ`)

### 시험 형식
- [ ] 100% 객관식
- [ ] 스택/큐 총정리표 암기
- [ ] 트레이싱(상태 변화 추적) 연습을 꼭 직접 손으로 해보기!
