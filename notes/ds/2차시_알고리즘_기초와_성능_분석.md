# 📘 2차시 — 알고리즘의 기초와 성능 분석 (시간·공간 복잡도, 빅오 표기법)

> 핵심 한 줄 요약
> 프로그램 = 자료구조 + 알고리즘. 성능은 시간과 공간으로 측정하며, 시간 복잡도는 데이터 크기 $n$에 따른 연산 횟수의 증가 추이를 본다. 이를 압축 표현한 것이 점근적 표기법(Big-O 등) 이다.

## 🎯 2차시 핵심 출제 포인트 프리뷰

1. 알고리즘의 5가지 필수 조건 중 유한성(Finiteness) → 4지 선다형 1문제 확정 출제 예고
2. 빅오 표기법(Big-O) 의 정의·구하는 법·종류 → 100% 출제
3. 시간 복잡도 계산: 기본 연산(산술·비교·대입·이동)만 카운트, 루프 제어는 제외
4. Polynomial vs Exponential: 컴퓨터로 해결 가능 vs 불가능의 경계
5. Worst Case: 빅오에서 부등호 $\leq$ 의 '$=$' 성립 시점

## 1. 자료구조와 알고리즘의 개요

### 1.1 프로그램 = 자료구조 + 알고리즘
- 프로그램의 정의: 컴퓨팅 프로그램으로 데이터를 처리하여 문제를 해결하는 것.
- 알고리즘 (Algorithm): 문제를 해결하는 방법론. 즉 데이터를 어떻게(How to) 처리할 것인지를 나타내는 명령어(Instruction)의 집합.
- 자료구조 (Data Structure): 처리 대상이 되는 데이터들이 잘 정리된 형태.
- 데이터 포함: 알고리즘은 단순히 절차만을 의미하는 것이 아니라, 작업의 대상이 되는 데이터까지 포함해야 한다.
- 효율성 (Efficiency): 컴퓨터를 사용하는 목적은 문제 해결을 잘(효율적으로) 하기 위함. 자료구조에서 "잘한다"는 것은 곧 "효율적이다" 를 의미.
- 성능의 2가지 측면: 시간(Time) 과 자원/공간(Resource/Space, 메모리).
    - 시스템의 요구사항(Context)에 따라 시간에 제한적인지, 자원에 제한적인지를 파악하여 적절한 알고리즘을 선택해야 함 (Trade-off 관계).

---

### 1.2 자료구조의 분류

자료구조는 크게 Primitive(기초 자료형) 와 Non-Primitive(파생 자료형/자료구조) 로 나뉜다.

#### 🔹 ① Primitive Data Type (기초 자료형)
프로그래밍 언어에서 기본적으로 제공하는 데이터 타입.

| 타입 | 크기 | 비고 |
|------|------|------|
| `char` | 1바이트 | 문자 |
| `int` | 보통 2 또는 4바이트 | 2바이트 기준: $-32{,}768 \sim 32{,}767$ / `unsigned int`: $0 \sim 65{,}535$ |
| `float` | 4바이트 | 실수 |
| `double` | 8바이트 | 실수 |

> 💡 교수님 언급: int 종류만 9개, 기초 자료형은 약 15개 정도 있음.

#### 🔹 ② 파생 자료형 (프로그래밍 언어 제공)

| 자료형 | 특징 |
|--------|------|
| 배열 (Array) | 같은 타입의 데이터들을 묶어 놓은 것 |
| 구조체 (Struct) | 다른 타입의 데이터들을 묶어 놓은 것 |
| 공용체 (Union) | 메모리 효율을 위해 여러 타입 중 가장 큰 사이즈를 기준으로 메모리를 할당하여 공유 |
| 포인터 (Pointer) | 메모리 주소가 저장되는 변수 (C언어에서 강력함) |
| 열거형 (Enum) | 직관적으로 알기 쉽도록 문자열을 정수로 처리 (예: Sunday=0, Monday=1) |

#### 🔹 ③ Non-Primitive Data Structure (자료구조)
데이터들의 관계(Relationship) 에 따라 선형(Linear)과 비선형(Non-Linear)으로 구분.

[선형 자료구조] — 1:1 관계, 순서가 중요. 내 뒤에 누가 오는지(Insertion·Deletion의 위치)에 따라 종류가 나뉨.

| 자료구조 | 특징 | 비유 |
|----------|------|------|
| 리스트 (List) | 앞·중간·뒤 어디서나 삽입/삭제 가능 | 할 일 목록 |
| 스택 (Stack) | 한쪽 끝(Top)에서만 삽입·삭제. LIFO | 쌓아놓은 접시 |
| 큐 (Queue) | 한쪽(Rear) 삽입, 반대쪽(Front) 삭제. FIFO | 프린터 대기열 |

[비선형 자료구조] — 1:N 또는 N:M. 순서보다 연결(Connection) 과 방문 순서(Traversal) 가 중요.

- 트리 (Tree)
    - 계층적(Hierarchical) 구조. 사이클 없음(Acyclic), 모든 노드 연결됨(Connected), 루트(Root) 존재.
    - 노드가 $N$개일 때 엣지는 $N-1$개.
    - 검색(Search) 을 위해 존재.
    - 탐색 방법(Traversal): 전위(Pre), 중위(In), 후위(Post), 레벨(Level) 4가지.
    - 비유: 가계도, 조직도.
- 그래프 (Graph)
    - 정점(Vertex/Node)과 간선(Edge)의 집합. 방향성·가중치(Weight) 가능.
    - 탐색 방법:
        - DFS (깊이 우선): 갈 수 있을 때까지 깊게 들어갔다가 돌아오는 방식. (예: A-B-C-E-D-F)
        - BFS (너비 우선): 인접 노드부터 먼저 방문. (예: A-B-C-D-E-F)
    - 스패닝 트리 (Spanning Tree): 모든 노드를 포함하면서 사이클이 없는 트리 ($N-1$개의 엣지).
    - 최소 스패닝 트리 (MST): 가중치의 합이 최소가 되는 스패닝 트리. (예: 통신망 구축 비용 최소화)

## 2. 알고리즘 (Algorithm)

### 2.1 알고리즘의 정의
특정 작업을 수행하는 유한한 명령어(Instruction)들의 집합.
- 데이터까지 포함하는 개념이며, 자료구조와 떼려야 뗄 수 없는 관계.

### 2.2 💡 [시험 출제 100%] 알고리즘의 필수 조건 5가지

> 교수님께서 별표를 치며 강조하신 부분. 4지 선다형 1문제 확정 출제 예고.

| 조건 | 영문 | 의미 |
|------|------|------|
| 입력 | Input | 외부에서 제공되는 자료가 0개 이상 (없어도 됨!) |
| 출력 | Output | 반드시 1개 이상 있어야 함 (없으면 알고리즘이 아님) |
| 유한성 💡 | Finiteness | 한정된 수의 단계 후에 반드시 종료되어야 함 (무한 루프 불가) |
| 명확성 | Definiteness | 각 명령어는 모호하지 않고 명확해야 함 |
| 유효성 | Effectiveness | 각 명령어는 실제로 실행 가능한 연산이어야 함 |

> ⭐ 교수님 코멘트: "유한이라는 걸 굉장히 중요하게 생각한다."

---

### 2.3 알고리즘의 기술 방법

| 방법 | 장점 | 단점 |
|------|------|------|
| 자연어 (Natural Language) | 인간이 읽기 쉬움 | 의미가 모호해질 수 있음 (ChatGPT 프롬프트처럼 구체적이어야 함) |
| 흐름도 (Flow Chart) | 화살표로 순서가 직관적 | 복잡해지면 매우 길어지고 이해 어려움 |
| 유사 코드 (Pseudo-code) ⭐ | 구조적이면서도 언어 문법에 얽매이지 않음. 핵심 로직에 집중 | — |
| 프로그래밍 언어 (C, Python) | 가장 정확 | 너무 구체적이어서 핵심 로직 파악을 방해 |

> 💡 Pseudo-code 특징: 대입 연산자는 `←` 화살표 사용. 특정 언어의 문법(변수 선언 등)에 얽매이지 않고, 구체적인 구현 세부 사항을 감춰줌.

## 3. 추상 데이터 타입 (ADT: Abstract Data Type)

### 3.1 ADT의 개념
- 데이터 타입: 데이터의 종류 (데이터 = 처리의 대상이 되는 모든 것).
- 추상화 (Abstraction): 사용자에게 중요한 핵심 정보(구조, 동작) 만 강조하고, 구현 세부 사항은 제거(Information Hiding, 정보 은닉) 하는 것.
- ADT의 정의: 데이터 타입을 수학적/추상적으로 정의한 것 → 자료구조의 '사용 설명서' 역할.

### 3.2 ADT의 구성 요소

| 구성 요소 | 설명 |
|----------|------|
| Data Value | 데이터의 값에 대한 설명 |
| Data Relationship | 자료들 간의 관계 (객체로서 묘사됨) |
| Operation | 객체들 사이의 연산 (Insertion, Deletion, Traverse, Is-Empty 등). 외부와의 인터페이스 역할 |

### 3.3 💡 [핵심 포인트] What vs How
- What에 집중: 추상화(Abstract)는 '무엇(What)' 을 하는지에 집중.
    - 예: "인서트를 마지막에 한다", "양쪽에서 한다"
- How는 배제: '어떻게(How)' 구현하는지는 설명하지 않음.
- 이유: 구체적인 프로그래밍 언어나 문법(그래머)까지 공개하면 알고리즘의 본질(핵심 내용) 에 집중하기 어렵기 때문.
- 그래서 우리는 수도코드(Pseudo-code) 를 사용한다.

## 4. 알고리즘의 성능 분석 개요

알고리즘의 성능은 크게 시간(Time) 과 공간/자원(Space/Memory) 두 측면에서 평가한다.

- Trade-off 관계: 메모리를 적게 쓰려면 수행 시간이 길어지고, 수행 시간을 단축하려면 메모리를 많이 써야 하는 경우가 많음.
- 내가 개발하려는 시스템의 요구사항(Context, 시간 제한적인지 메모리 제한적인지)에 맞춰 적절한 알고리즘을 선택해야 함.
- 현대에는 자원보다 시간이 훨씬 제한적인 요소.

### 4.1 성능 측정 방법 두 가지

#### ① 실제 수행 시간 측정
- 방법: C언어의 `clock()` 함수 등을 사용하여 시작·종료 클럭 차이 계산.
    $$\text{duration} = \frac{\text{finish} - \text{start}}{\text{CLOCKS\_PER\_SEC}}$$
- 한계: 동일한 하드웨어 환경에서 모두 구현해봐야 하므로 복잡하고 거대한 시스템에서는 비현실적.

#### ② 알고리즘의 복잡도 분석 (Complexity Analysis) ⭐
- 개념: 코드 구현 없이 알고리즘이 수행하는 명령어(Instruction) 연산 횟수를 수학적 함수로 나타냄.
- 교수님 비유: "지하철을 타고 A에서 B까지 갈 때 꼭 끝까지 가봐야 시간을 아나요? 3정거장에 5분 걸린다는 걸 알면, 10정거장이면 대략 얼마나 걸릴지 예측(분석) 할 수 있죠."
- 시간 복잡도 (Time Complexity) $T(n)$: 수행 시 필요한 연산 횟수를 입력 크기 $n$에 대한 함수로 표현.
- 공간 복잡도 (Space Complexity) $S(n)$: 수행 시 필요한 메모리 크기를 입력 크기 $n$에 대한 함수로 표현.

## 5. 시간 복잡도 함수 $T(n)$ 구하기

우리가 알고 싶은 것은 구체적인 "1분, 2초"가 아니라, 데이터 크기 $n$이 증가함에 따라 실행 시간이 얼마나 급격하게 증가하는지(성장률, 증가 추이) 이다.

### 5.1 연산 횟수 카운트 규칙
- 기본 연산만 고려: 산술, 대입, 비교, 이동 연산의 수행 횟수만 카운트.
- 루프 제어 연산 제외: `for` 문의 인덱스 증가 `i++` 같은 루프 제어용 연산은 카운트에서 제외.
    - 단, 루프 내부의 연산이 몇 번 반복되는지는 곱해줘야 함.

### 5.2 연산 횟수 계산 예시 — 최대값 찾기 알고리즘

의사코드:
```
temp ← A[0]
for i ← 1 to n-1 do
    if A[i] > temp then
        temp ← A[i]
return temp
```

데이터가 $n$개일 때의 연산 횟수:

| 명령어 | 연산 종류 | 횟수 |
|--------|-----------|------|
| `temp = A[0]` | 대입 | $1$번 |
| `A[i] > temp` | 비교 (루프 내) | $n-1$번 |
| `temp = A[i]` | 이동/대입 (최악: 매번 갱신) | $n-1$번 |
| `return temp` | 반환 | $1$번 |

$$T(n) = 1 + (n-1) + (n-1) + 1 = 2n$$

### 5.3 알고리즘 간의 성능 비교 (교수님 예시)

- 알고리즘 A: $T(n) = 3n + 2$
- 알고리즘 B: $T(n) = 5n^2 + 6$

| $n$ | 알고리즘 A | 알고리즘 B |
|-----|-----------|-----------|
| 10 | 32 | 506 |
| 100 | 302 | 50,006 |

> 💡 결론: 데이터 크기 $n$이 커질수록 최고차항의 차이가 결과에 지대한 영향을 미치며, 격차가 급격히 벌어진다. → 그래서 최고차항만으로 분석(점근적 표기법)해도 충분하다!

## 6. 점근적 표기법 (Asymptotic Notation) 💡

데이터 개수가 많아지면 $T(n)$에서 가장 차수가 큰 최고차항이 결과에 압도적인 영향을 미치고 나머지 항이나 계수는 무시 가능. 이를 간단히 표기하는 방법이 점근적 표기법이다.

### 6.1 💡 [시험 출제 100%] 빅오 표기법 (Big-O Notation, $O$)

#### 정의
두 함수 $f(n)$과 $g(n)$이 있을 때, 모든 $n \geq n_0$에 대하여
$$|f(n)| \leq c \cdot |g(n)|$$
을 만족하는 2개의 상수 $c$와 $n_0$가 존재하면, $f(n) = O(g(n))$이다.

#### 의미
- 상한(Upper Bound): "아무리 최악의 상황이라도 이 함수보다는 연산 횟수가 적거나 같다."
- 입력의 크기가 커질 때, 연산 횟수는 아무리 많아도 이 함수에 계수를 곱한 값보다 같거나 작다.

#### 💡 Worst Case (최악의 경우)
- 교수님 매우 강조: 부등호의 '같다($=$)'가 성립하는 시점이 바로 최악의 경우(Worst Case).
- 교수님 비유: "인터넷 뱅킹을 할 때 최악의 경우 15ns면 리스폰스를 받을 수 있다. 13ns가 될 수도 있지만 최악의 경우 15ns를 보장한다."

#### 구하는 법
시간 복잡도 함수에서 최고차항만 남기고 계수는 뗀다.
$$T(n) = 2n^2 + 3n + 1 \;\;\longrightarrow\;\; O(n^2)$$
$$T(n) = n^2 + n + 1 \;\;\longrightarrow\;\; O(n^2)$$

### 6.2 빅 오메가 ($\Omega$) & 빅 세타 ($\Theta$)

| 표기 | 의미 | 부등식 | 관련 개념 |
|------|------|--------|-----------|
| $\Omega$ (빅 오메가) | 하한(Lower Bound) | $f(n) \geq c \cdot g(n)$ | Best Case (예: 순차 탐색에서 값이 맨 앞에 있을 때) |
| $\Theta$ (빅 세타) | 상한과 하한 동시에 | $c_1 g(n) \leq f(n) \leq c_2 g(n)$ | 계수가 2개 필요하여 구하기 어려워 잘 쓰지 않음 |
| $O$ (빅오) | 상한(Upper Bound) | $f(n) \leq c \cdot g(n)$ | Worst Case |

---

### 6.3 빅오 표기법의 종류 및 실행 시간 비교

효율적인 순서대로 (좋은 → 나쁜):

| 표기법 | 명칭 | 특징 |
|--------|------|------|
| $O(1)$ | 상수형 | 데이터가 증가해도 연산 횟수 일정. (교수님 농담: "얼굴도 예쁜데 성격도 완벽한 사람처럼 현실엔 거의 없는 이상적인 케이스") |
| $O(\log n)$ | 로그형 | 증가 추세가 매우 완만. (예: 이진 탐색) |
| $O(n)$ | 선형 | 데이터에 비례해 증가. (1중 루프) |
| $O(n \log n)$ | 로그선형 | 중첩 루프에서 바깥은 1씩, 안쪽은 2배씩(지수형태) 증가할 때 |
| $O(n^2)$ | 2차형 | 2중 루프 |
| $O(n^3)$ | 3차형 | 3중 루프 |
| $O(2^n)$ | 지수형 | 매우 비효율적 |
| $O(n!)$ | 팩토리얼형 | 극도로 비효율적 |

---

### 6.4 💡 [매우 중요] Polynomial vs Exponential

| 분류 | 범위 | 해결 가능성 |
|------|------|------------|
| Polynomial (다항식 시간) | $O(1) \sim O(n^3)$ (넓게는 $O(n^4)$까지) | 컴퓨터로 해결 가능한 마지노선 |
| Exponential (지수/팩토리얼 시간) | $O(2^n), O(n!)$ | 알고리즘으로 해결 비효율적·사실상 불가 |

> ⭐ 교수님 왈: "루프 3개까지는 봐준다."

#### 💡 [Q&A 지엽 출제 포인트] 4차 함수와 지수/팩토리얼형
- 학생 질문: "3차까지만 컴퓨터로 하고 4차는 수작업이 빠르나요?"
- 교수님 답변: "요즘엔 컴퓨팅 파워가 좋아져서 4차 함수($O(n^4)$)도 폴리노미얼(다항식형) 이므로 컴퓨터로 해결 가능합니다. 하지만 지수형($O(2^n)$)과 팩토리얼형($O(n!)$) 은 알고리즘으로 문제를 해결할 수 없을 정도로 비효율적이라 차라리 수작업이 낫습니다."

---

## 7. 공간 복잡도 (Space Complexity)

### 7.1 정의
프로그램을 실행할 때 메모리를 얼마나 사용하는지 보여주는 함수 $S(n)$.

### 7.2 구성 요소

| 영역 | 설명 |
|------|------|
| 고정 영역 | 입력 크기와 무관. 프로그램 자체나 단순 변수가 차지. (프로그램 영역 $P$, 기본 변수 등) |
| 가변 영역 | 입력 개수 $n$에 따라 변하는 공간 (예: 크기 $n$인 배열) |

### 7.3 계산식 예시
$$S(n) = n + P + 3$$
(크기 $n$ 배열 + 프로그램 영역 $P$ + 기본 변수 3개)

---

### 7.4 💡 메모리 사용 구조 (위에서 아래로 쌓임)

| 순서 | 영역 | 설명 |
|------|------|------|
| 1 | OS (Operating System) | 운영체제 |
| 2 | Program (코드 영역) | 프로그램 코드 — 고정 |
| 3 | Global / Static Variable | 전역/정적 변수 — 고정 (Static Allocation) |
| 4 | Heap (힙 영역) | 동적 할당 (Dynamic Allocation). 데이터 크기에 따라 변함 |
| 5 | Stack (스택 영역) | 함수 호출 시 Stack Frame이 쌓임 |

#### Stack Frame의 구성 요소
1. Return Address (돌아갈 주소)
2. Local Variable (지역 변수)
3. Parameter (매개 변수)

---

## 8. 최선, 평균, 최악의 경우

알고리즘의 수행 시간은 입력 데이터의 상태(집합) 에 따라 크게 달라질 수 있다.

| 경우 | 의미 | 예시 (순차 탐색) | 표기 | 중요도 |
|------|------|------------------|------|--------|
| Best Case (최선) | 수행 시간이 가장 빠름 | 찾는 값이 맨 앞 | $O(1)$ | 큰 의미 없음 |
| Average Case (평균) | 모든 상황 평균 | — | — | 수학적 계산이 까다로움 |
| Worst Case (최악) 💡 | 수행 시간이 가장 오래 걸림 | 찾는 값이 맨 뒤 or 없음 | $O(n)$ | 가장 널리, 중요하게 사용 |

> ⭐ Worst Case가 중요한 이유: 비행기 관제, 의료 기기처럼 최악의 상황에서도 제한 시간 내 동작을 보장해야 하는 시스템이 많기 때문.

## 🎯 시험 직전 최종 암기 체크리스트

- [ ] 알고리즘 5대 조건 중 입력은 0개 이상 / 출력은 1개 이상 / 유한성 필수
- [ ] 빅오 정의 수학적 공식 ($|f(n)| \leq c \cdot |g(n)|, \; n \geq n_0$)
- [ ] 시간 복잡도 계산 — 루프 제어 연산 제외, 내부 연산만 세고 곱함
- [ ] $O(n^3)$까지는 폴리노미얼 (해결 가능), $O(2^n)$, $O(n!)$은 비효율
- [ ] Worst Case는 빅오의 등호 성립 시점
- [ ] 메모리 영역: Heap = 동적, Stack = 함수 호출, Stack Frame = Return Addr + Local Var + Parameter
- [ ] $T(n) = 2n^2 + 3n + 1 \;\to\; O(n^2)$ (최고차항만, 계수 뗌)
